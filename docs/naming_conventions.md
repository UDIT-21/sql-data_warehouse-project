This document outlines the naming conventions used for schemas, tables, views, columns, stored procedures, and other objects in the Data Warehouse.

## General Principles

* **Naming Conventions:** Use `snake_case`, with lowercase letters and underscores (`_`) to separate words.
* **Language:** Use English for all names.
* **Avoid Reserved Words:** Do not use SQL reserved words (e.g., `SELECT`, `TABLE`, `DATE`) as object names.
* **Schema Usage:** All objects must reside in their respective layer's schema (`bronze`, `silver`, or `gold`).

## Table Naming Conventions

### Bronze Rules
All tables in the Bronze layer must start with the source system name, and table names must match their original names from the source without renaming or translating business logic.

* **Format:** `<sourcesystem>_<entity>`
* `<sourcesystem>`: Name of the source system (e.g., `crm`, `erp`).
* `<entity>`: Exact table name or file name from the source system.

**Examples:**
* `crm_cust_info` → Customer information extracted directly from the CRM system.
* `erp_loc_a101` → Location data extracted directly from the ERP system.

### Silver Rules
Tables in the Silver layer retain the same naming structure as the Bronze layer to ensure clear traceability and lineage back to the source system. 

* **Format:** `<sourcesystem>_<entity>`
* `<sourcesystem>`: Name of the source system (e.g., `crm`, `erp`).
* `<entity>`: Exact table name from the source system.

**Examples:**
* `crm_cust_info` → Cleansed and standardized customer information from the CRM system.
* `erp_px_cat_g1v2` → Cleansed product category data from the ERP system.

### Gold Rules
All views/tables in the Gold layer must use meaningful, business-aligned names, starting with a category prefix to denote their role in the Star Schema. Raw source abbreviations (like `cust_info` or `az12`) are strictly avoided.

* **Format:** `<category>_<entity>`
* `<category>`: Describes the role of the object, such as `dim` (dimension) or `fact` (fact table).
* `<entity>`: Descriptive, pluralized name of the table, aligned with the business domain.

**Examples:**
* `dim_customers` → Dimension table for integrated customer data.
* `fact_sales` → Fact table containing transactional sales data.

#### Glossary of Category Patterns

| Pattern | Meaning | Example(s) |
| :--- | :--- | :--- |
| `dim_` | Dimension table (business context/attributes) | `dim_customers`, `dim_products` |
| `fact_` | Fact table (measurable, transactional events) | `fact_sales` |
| `report_` | Flat report table for specific BI dashboard needs | `report_sales_monthly` |

## Column Naming Conventions

### Surrogate Keys
All primary keys generated within the data warehouse for dimension tables must use the suffix `_key`. This distinguishes them from natural/business keys sourced from the CRM or ERP.

* **Format:** `<entity>_key`
* `<entity>`: Refers to the name of the dimension or entity the key belongs to.
* `_key`: A suffix indicating that this column is a system-generated surrogate key.

**Examples:**
* `customer_key` → Surrogate key in the `dim_customers` view.
* `product_key` → Surrogate key in the `dim_products` view.

### Technical Columns
All technical metadata columns generated by the ETL process must start with the prefix `dwh_`, followed by a descriptive name indicating the column's purpose.

* **Format:** `dwh_<column_name>`
* `dwh_`: Prefix exclusively reserved for data warehouse system-generated metadata.
* `<column_name>`: Descriptive name indicating the column's purpose.

**Examples:**
* `dwh_create_date` → System-generated column used to store the timestamp when the record was inserted into the Silver layer.

## Stored Procedure Conventions

All stored procedures used for ETL (Extract, Transform, Load) operations must clearly indicate the action and the target destination layer.

* **Format:** `load_<layer>`
* `<layer>`: Represents the data warehouse layer being populated.

**Examples:**
* `load_bronze` → Stored procedure for truncating and bulk inserting data into the Bronze layer.
* `load_silver` → Stored procedure for cleansing, transforming, and loading data into the Silver layer.
