# 🏭 Modern Data Warehouse — SQL Server + Medallion Architecture

> An end-to-end **Data Engineering** portfolio project implementing a production-grade Data Warehouse on SQL Server.  
> Raw CSV files are transformed into a business-ready Star Schema through a clean, layered **Medallion Architecture** (Bronze → Silver → Gold).

Built as a self-taught project alongside the **Data With Baraa** series. Every script, document, and pattern here reflects real industry practice — the same workflows used by data engineers at scale.

## 🎓 What This Project Teaches

This project was built to deeply internalize the patterns that repeat across every data warehouse, regardless of the dataset. If you follow it end to end — writing each script yourself — you will come away with hands-on experience in:

### Data Engineering Fundamentals
- How a **Medallion Architecture** separates concerns across Bronze, Silver, and Gold layers
- Why **physical tables** are used for Bronze/Silver (persistence, debugging) but **Views** for Gold (always-live, zero storage cost)
- The difference between **natural keys** (from source systems) and **surrogate keys** (generated by the warehouse)
- What a **Star Schema** is, why it outperforms normalized schemas for analytics, and how to build one from scratch

### T-SQL & SQL Server Skills
- `BULK INSERT` from CSV files with `FIRSTROW`, `FIELDTERMINATOR`, and `TABLOCK`
- `TRUNCATE` vs `DELETE` and why ETL always uses TRUNCATE for full loads
- `CREATE OR ALTER PROCEDURE` — stored procedure authoring with `BEGIN TRY / BEGIN CATCH`
- `DATEDIFF`, `GETDATE()`, `DATETIME2 DEFAULT GETDATE()` for timing and audit columns
- `ROW_NUMBER() OVER (PARTITION BY ... ORDER BY ...)` for deduplication
- `LEAD()` window function for deriving SCD Type 2 end dates
- `NULLIF()` to prevent division-by-zero errors
- `ISNULL()` for safe NULL replacement
- `CASE WHEN` for decoding raw codes into readable labels
- Integer-to-DATE casting pattern for `YYYYMMDD` source data
- `INFORMATION_SCHEMA.TABLES` and `INFORMATION_SCHEMA.COLUMNS` for metadata queries
- `COALESCE()` for multi-source data resolution (e.g., CRM gender vs ERP gender)

### Data Quality & Governance
- Writing systematic **quality check scripts** for every layer (row counts, structure checks, sample previews, business rule validation)
- Validating referential integrity between Fact and Dimension tables
- Detecting and handling NULLs, duplicates, whitespace, invalid dates, and inconsistent measures
- Enforcing **naming conventions** consistently (`snake_case`, `dim_` / `fact_` prefixes, `dwh_` audit prefix, `_key` suffix for surrogate keys)
- Building a professional **Gold Layer Data Catalog** documenting every object, column, data type, and business description

### Data Modeling & Analysis Concepts
- **Data Integration**: combining CRM and ERP systems with mismatched key formats
- **Data Lineage**: tracing a column from its raw CSV source all the way to a Gold Layer metric
- **SCD Type 2** (Slowly Changing Dimensions) — tracking historical versions of product records
- **Exploratory Data Analysis (EDA)** across six structured phases: database exploration, dimension profiling, date analysis, measures aggregation, magnitude analysis, and ranking analysis
- **KPI identification** — distinguishing a raw metric from a business Key Performance Indicator
- Basket analysis, Customer Lifetime Value proxy, customer activation rate, and average order value

## 🏗️ Architecture & Data Flow

```
[CRM CSV Files]   [ERP CSV Files]
        │                │
        └────────┬────────┘
                 ▼
          ┌─────────────┐
          │  🥉 BRONZE  │  Raw ingestion — no transformation, full source fidelity
          └──────┬──────┘
                 ▼
          ┌─────────────┐
          │  🥈 SILVER  │  Cleansed, standardized, deduplicated, type-cast
          └──────┬──────┘
                 ▼
          ┌─────────────┐
          │   🥇 GOLD   │  Star Schema Views — business-ready, always live
          └──────┬──────┘
                 ▼
        [BI / Analytics / EDA]
```

### Layer Responsibilities

| Layer      | Storage Type     | Purpose                                  | Refresh Method             |
|------------|------------------|------------------------------------------|----------------------------|
| **Bronze** | Physical Tables  | Land raw data exactly as it arrives      | TRUNCATE + BULK INSERT     |
| **Silver** | Physical Tables  | Clean, decode, deduplicate, type-cast    | TRUNCATE + INSERT...SELECT |
| **Gold**   | Views (no data)  | Star Schema — shapes data for the business | Automatic (reads Silver)  |

> **Why Views for Gold?** When Silver is refreshed, Gold instantly reflects the change — no extra step, no stale data, no storage cost. Views are the correct pattern for a presentation layer that doesn't transform data, only shapes it.

## 🔗 Source Data & Integration

This warehouse integrates two independent source systems. Neither uses the same key format — resolving that is the core data engineering challenge of this project.

**CRM System** (front office — sales & customer transactions)
- `crm_cust_info` → core customer attributes
- `crm_prd_info` → product master with historical versions
- `crm_sales_details` → transactional sales orders

**ERP System** (back office — enrichment & reference data)
- `erp_cust_az12` → customer demographics (birthdate, gender)
- `erp_loc_a101` → customer country mapping
- `erp_px_cat_g1v2` → product category/subcategory reference

## ⚙️ Silver Transformations — The Five Patterns

Every transformation written in the Silver layer falls into one of five reusable patterns.

| # | Pattern | Problem Solved | Function Used |
|---|---------|----------------|---------------|
| 1 | **TRIM** | Whitespace in string fields from human-entered source data | `TRIM()` |
| 2 | **CASE WHEN Decode** | Single-char codes (`'S'`, `'M'`, `'F'`) → readable labels | `CASE WHEN` + `UPPER(TRIM())` |
| 3 | **ROW_NUMBER Dedup** | Multiple versions of same customer — keep only the latest | `ROW_NUMBER() OVER (PARTITION BY ... ORDER BY DESC)` |
| 4 | **Integer Date Cast** | Dates stored as `YYYYMMDD` integers — convert to real `DATE` | `CAST(CAST(x AS VARCHAR) AS DATE)` with length validation |
| 5 | **Business Rule Recalc** | `sales ≠ quantity × price` — recalculate from trusted fields | `CASE WHEN` + `NULLIF()` |

**Bonus — SCD Type 2 with `LEAD()`:** Product records change over time. `LEAD()` peeks at the next version's start date to derive the current version's end date, creating a full historized dimension automatically.

## 🥇 Gold Layer — Star Schema

Three objects form the complete Star Schema:

```
           ┌──────────────────┐
           │  dim_customers   │
           │  (customer_key)  │
           └────────┬─────────┘
                    │ customer_key
                    ▼
┌───────────────────────────────────┐
│             fact_sales            │
│  order_number · product_key       │
│  customer_key · order_date        │
│  sales_amount · quantity · price  │
└───────────────────┬───────────────┘
                    │ product_key
                    ▼
           ┌──────────────────┐
           │  dim_products    │
           │  (product_key)   │
           └──────────────────┘
```

**`gold.dim_customers`** — Integrates CRM + ERP demographics + ERP location. CRM is the trusted gender source; ERP is fallback only.

**`gold.dim_products`** — Integrates CRM product master + ERP categories. Filters to active products only (`prd_end_dt IS NULL`).

**`gold.fact_sales`** — Transactional grain (one row per order line). Stores surrogate keys + measures only. All descriptive context lives in the dimensions.

> **Join Rule:** Always join on `customer_key` / `product_key` (surrogate keys). Never join on `customer_id` or `product_id` — those are source-system natural keys and are not guaranteed unique in the dimensions.

## 📊 Exploratory Data Analysis — Six Phases

The EDA script (`scripts/data_analysis/exploratory_data_analysis.sql`) runs after Gold is verified and before any reporting. It is structured as a stakeholder conversation with the data:

| Phase | Question Being Answered |
|-------|------------------------|
| **1 — Database Exploration** | What objects exist? What columns do they expose? |
| **2 — Dimensions Exploration** | What are the distinct values? Any misspellings or NULL categories? |
| **3 — Date Exploration** | How many years of history? What is the customer age range? |
| **4 — Measures Exploration** | What are the headline KPIs — total revenue, units, orders, AOV? |
| **5 — Magnitude Analysis** | Which countries, categories, and customers drive the business? |
| **6 — Ranking Analysis** | Top 5 products, top 10 customers, bottom 5 products — who are the outliers? |

## 🚀 How to Run the Project

### Prerequisites
- SQL Server Express (any recent version)
- SQL Server Management Studio (SSMS)
- Git

### Step-by-Step Execution

Run scripts in this exact order. Each step depends on the previous one completing successfully.

```
Step 1 → scripts/create_database.sql
         Creates the DataWarehouse database and bronze/silver/gold schemas.
         ⚠️  WARNING: Drops the database if it already exists. Run once only.

Step 2 → scripts/bronze/create_bronze_layer.sql
         Creates all 6 bronze tables (raw structure mirrors source CSVs).

Step 3 → scripts/bronze/loaddata_bronze_layer.sql
         Runs: EXEC bronze.load_bronze
         Loads all 6 CSV files via BULK INSERT. Update file paths to your local drive.

Step 4 → scripts/bronze/verify_bronze_layer.sql
         Confirms row counts > 0, structure correct, sample data looks right.
         Do not proceed if any table is empty.

Step 5 → scripts/silver/create_silver_layer.sql
         Creates all 6 silver tables (cleaned types + dwh_create_date audit column).

Step 6 → scripts/silver/loaddata_silver_layer.sql
         Runs: EXEC silver.load_silver
         Applies all transformation patterns — dedup, decode, date cast, recalc.

Step 7 → scripts/silver/verify_silver_layer.sql
         Runs 18 quality checks. All expected results are documented inline.
         Investigate any check that returns unexpected rows before continuing.

Step 8 → scripts/gold/create_gold_layer.sql
         Creates gold.dim_customers, gold.dim_products, gold.fact_sales as Views.

Step 9 → scripts/gold/verify_gold_layer.sql
         Validates referential integrity (0 orphaned rows expected) + business logic.

Step 10 → scripts/data_analysis/exploratory_data_analysis.sql
          Run phase by phase. Read the inline commentary — each query explains
          what business question it answers and what the result implies.
```

### Update File Paths Before Running Bronze Load

In `loaddata_bronze_layer.sql`, update the `FROM` path in each `BULK INSERT` block to match your local machine:

```sql
-- Change this:
FROM 'D:\DataWarehouse Project\Dataset\source_crm\cust_info.csv'

-- To your actual path, e.g.:
FROM 'C:\Users\YourName\data-warehouse-project\datasets\source_crm\cust_info.csv'
```
